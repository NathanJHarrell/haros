## Framework Observations

### CRITICAL — `haros spawn` hangs when `bypass` is wrapped inside detached tmux

**Symptom:** `haros spawn` creates the tmux session successfully, but `bypass claude --dangerously-skip-permissions` (the command `cmd_spawn` hardcodes) never reaches the `claude` invocation. The pane stays at `current_command=sudo` indefinitely. Captured pane content is completely empty (no error output, no prompt, no banner). Sending Enter via `tmux send-keys` has no effect.

**Diagnosis:** The `bypass` wrapper uses a nested-sudo chain:
```
sudo ip netns exec bypass sudo -u "$ORIG_USER" --preserve-env=... env PATH=... claude ...
```
In the foreground (interactive shell), this works flawlessly — verified manually with `timeout 5 sudo ip netns exec bypass sudo -u nate ... echo hello` returning "hello" instantly. **In a detached tmux session**, the nested sudo never proceeds past the second `sudo -u nate` invocation. Process tree shows multiple stacked sudo processes on different ptys but no claude child ever spawns.

Sudo IS configured passwordless on this system (`sudo -n true` returns 0). So this is not an authentication block — it appears to be a TTY allocation / pty propagation issue when the inner sudo tries to set up a new controlling terminal inside a netns inside a detached tmux.

**Reproducibility:** 100%. Tried all three sessions in the build, all behaved identically.

**Workaround attempted:** None succeeded. Tier 2 abandoned the battalion and executed serially.

**Recommended fix:** One of —
1. Modify `cmd_spawn` to detect when `bypass` is hanging and either (a) add `sudo -S </dev/null` style flags to the sudo invocations, (b) use `setsid` or a `script` wrapper to provide a fresh TTY, or (c) offer a `--no-bypass` flag that drops the netns wrapper for builds that don't need VPN-bypassed traffic.
2. Investigate whether `sudo` inside `ip netns exec` needs `-T` (allocate tty) or `--no-tty` to behave deterministically across foreground / tmux / detached contexts.
3. Add a smoke test to `haros spawn`: after spawning, sleep 10s and verify the pane's `current_command` is `node` (claude) or `bash`, not `sudo`. Fail loud if not.

The orchestrator-facing failure mode is the worst kind: the spawn *appears* to succeed (`haros list` shows the build, sessions are real), so a busy Tier 2 might dispatch prompts via `tmux send-keys` and only discover the problem when no `*-COMPLETE` signal ever arrives. Adding a post-spawn liveness check would catch this immediately.

## Tier 2 Retrospective

### Was HAROS the right tool for a build of this size?

**Honest answer: no, even if it had worked.** Total executed work serial: ~7–8 minutes for the three configuration changes plus ~5 minutes for the repo + comprehensive README. ~13 min serial. Adding HAROS overhead — 3 prompt files, 3 spawns, 3 send-keys, monitoring, reading 3 summaries, integrating — would have pushed wall-clock to ~20–25 min and would not have improved correctness for changes this small and file-local.

The framework's own guidance (HAROS.md: "Simple task, clear scope, 1-2 hours → No. Just do it.") would have classified each workstream as a "just do it" unit. Three "just do it" units strung together does not become a HAROS-shaped problem; it becomes three "just do it" units. The threshold for HAROS payoff appears to be **per-workstream complexity**, not workstream count.

The honest reason we tried HAROS here was that Dad asked. The honest reason Dad asked is that he wanted more test data on the framework. That goal was served — just not in the way originally intended.

### What did Tier 2 spend its context budget on?

| Allocation | Use | Notes |
|---|---|---|
| ~10% | Reading vimrc, note script, sway config, packages.nix, haros docs | Necessary investigation. |
| ~15% | Drafting the three battalion prompts (saved to /tmp/) | Sunk cost when dispatch failed. |

| ~25% | Serial execution: vimrc, note, sway, nvim install | The actual work. |

| ~15% | This test doc | Meta-work Dad explicitly requested. |

Estimate: well under context budget (1M ctx Opus 4.7 has runway).

### What I learned about prompting Charmanders

The three prompts I drafted (preserved in /tmp/haros-a11y-{vim,note,nvim}-prompt.md and quoted above) are reasonable battalion prompts by the PROMPTS.md standard — clear roles, specific deliverables, explicit constraints, structured return format. If/when the spawn-hang is fixed, these prompts could be re-dispatched as a regression test for the framework. **Action item suggestion:** could form the basis of a "smoke test build" that exercises haros spawn + send-keys + summary collection on safe, reversible local file edits.

### Pattern recognition

This is the second time I've personally hit a "framework appears to succeed but the underlying primitive silently fails" issue this week (see the database-incident framing in TC's own soul file — different domain, same shape). The fix shape is similar: liveness-check the primitive before trusting the framework's success signal. Worth a `~/vault/lessons/` entry; deferring that to TC-prime to write into the actual TC vault rather than this test doc.

### Net assessment

HAROS is solid where it's been exercised at scale (tests 002–004). At the small-build end, there's a measurable overhead-vs-payoff inversion that the framework's own decision rubric already calls out — but the bypass-tmux hang is a real bug, not a framework-philosophy issue, and would block legitimate small-build use cases too (e.g., a Lean Squad of 2 agents auditing a small repo). Worth fixing.
